# Gate 标准反思与解耦架构方向评估

**日期**：2026-07-12

## 1. PZ 的核心观察：门禁过严

PZ 的观察一针见血。WSBuffer 测试是最有说服力的证据：一篇已经被 FAST 录用的论文，在我们的标准下会得到 Weak Reject。这意味着我们的 kill gate 系统性地**过滤掉了可发表的工作**。

### 1.1 门禁过严的根因

回顾 23 次 Kill，我们的标准实际上是：

```
如果存在任何已有工作触及了核心声称的同一设计空间
→ Kill
```

这个标准的问题是：**它优化的是"零重叠"，而不是"可发表"。** 现实中，FAST/VLDB/SIGMOD 的录用标准是：

```
1. 问题是否真实且重要？
2. 故事是否清晰？（background → problem → why existing work doesn't solve it → solution）
3. 设计是否自洽？
4. 实验是否证实了声称？
5. 与 SOTA 相比是否有可测量的改善？
```

注意这里没有"核心机制必须完全没有先例"这一条。大量顶会论文使用已知的技术（B-tree、LSM、io_uring、coroutine、cache partitioning），贡献在于**将已知技术应用到一个被清晰刻画的新问题上，并通过系统设计产生显著改善**。

### 1.2 被过度 Kill 的方向回顾

用修正后的标准重新审视，以下被 Kill 的方向中至少有两个在更宽松但仍合理的标准下可能存活：

1. **软件栈开销方向**：IISWC 2025 证明了 24% SSD 利用率，我们的 P0 也显示 45% IOPS 利用率。这个方向**从未被正式 Kill**——它在 problem gate 阶段被 side-track 了。VeloANN 优化的是 CPU 利用率（协程重叠），不是 I/O 路径 per-I/O 开销；Turbocharging 用 io_uring 但针对 IVF 不是图索引。这些"类似工作"在严格标准下构成 Kill，但在合理标准下只是 related work，不是覆盖。

2. **Graph Cracking（查询驱动物理重组）**：CrackIVF (VLDB 2025) 只做 IVF。图索引的 cracking 是一个不同的问题（依赖图结构不是 partition 结构）。被 Kill 主要是因为我们认为"cracking 的概念已被 CrackIVF 覆盖"——但 CrackIVF 覆盖的是 IVF，不是 graph。

### 1.3 修正后的标准

建议将 kill 标准从"是否有重叠"改为：

```
1. 是否有直接等价的已发表系统？（不是"类似的"，是"解决完全相同问题的"）
2. 我们能否指出现有工作的一个具体的、可验证的不足？
3. 我们的设计是否能比现有最强 baseline 产生 ≥2× 可测量改善？
4. 故事是否自洽？（problem → motivation → design → evaluation 链条是否无断裂）
```

满足 1-4 的方向就值得做，即使个别技术组件（io_uring、cache partitioning、co-location）是已知的。

## 2. 解耦架构方向评估

### 2.1 什么是"解耦架构"

在驻盘图 ANN 系统中，"解耦架构"指将以下数据分开存储：

- **拓扑（邻接表）**：graph 结构，决定搜索路径
- **坐标向量**：用于精确距离计算
- **PQ 编码**：用于近似距离计算

代表：DGAI（分离存储），vs. DiskANN/OdinANN（耦合：vertex record = neighbors + coordinates + PQ 一体）

### 2.2 解耦架构的已知优势

1. **拓扑页密度高**：不需要存储坐标，每页可以放更多邻居 → beam search 每次 I/O 获取更多拓扑信息
2. **灵活性**：拓扑和坐标可以独立管理、独立更新
3. **拓扑鲁棒性（A0 发现）**：拓扑对坐标变化具有强鲁棒性 → 支持解耦
4. **PQ 代码可以和拓扑共置**：不需要额外 I/O 就能算近似距离

### 2.3 解耦架构的可验证问题

**问题一：Reranking I/O 放大。** beam search 找到候选后，必须从坐标存储读取精确坐标做 reranking。这些候选在坐标存储中随机分布 → 随机 4KB 读。耦合架构不需要这一步。

**问题二：I/O 路径序列化。** 拓扑读 → PQ 近似 → 坐标读 → 精确计算。这是一个串行链。在耦合架构中，一次读就拿到了一切。

**问题三：缓存干扰。** 拓扑页和坐标页竞争 page cache / device buffer。热拓扑页被冷坐标页驱逐可能降低命中率。

**问题四：现代 SSD 下的 trade-off 位移。** 当 SSD 带宽从 500MB/s 增长到 7GB/s，"多读一些字节但少做一次 I/O"的 trade-off 发生变化。高带宽下，较大的 I/O（读整个 coupled record）的相对开销降低，而多次小 I/O 的 per-I/O 固定开销（syscall、中断、调度）变成瓶颈。

### 2.4 可能的论文故事

**故事核心**：现有解耦架构是在低带宽 SSD 时代设计的——拆分存储以减少每次 I/O 读取的无用字节。但在高带宽 NVMe 时代，per-I/O 固定开销（软件栈、中断、调度）成为主导，解耦带来的额外 I/O 次数反而成为瓶颈。

**论文结构**：
1. **Characterization（贡献一）**：系统性 profiling 解耦 vs 耦合架构在现代 NVMe 上的 I/O 行为。量化 reranking I/O 放大、缓存干扰、per-I/O 软件开销。
2. **问题定义**：在高带宽 SSD 上，解耦架构的 I/O 次数开销超过了它节省的 I/O 字节收益。
3. **系统设计（贡献二）**：一种混合架构（或者说针对解耦架构的 I/O 优化方案），既保持解耦的灵活性，又减少 I/O 次数。具体机制可以包括：
   - 自适应 co-location：将高频共访问的拓扑和坐标页物理邻接
   - 批量 I/O：利用 beam search 的可预测性，提前批量提交坐标读请求
   - 基于 PQ 分数的投机预取：PQ 分数高的候选，提前发起坐标读
4. **评估（贡献三）**：在多个数据集和硬件配置上验证改善。

### 2.5 这个方向会被 Kill 吗？

用**旧标准**：可能会被 Kill，因为 VeloANN 做了协程+异步 I/O、LIOS 做了 I/O/computation overlap、DiskANN 已经是耦合架构。

用**修正标准**检查：
1. **有无直接等价系统？** 没有。没有系统专门研究"解耦架构在高带宽 SSD 上的 I/O 放大问题"。VeloANN/LIOS 优化 CPU 利用率，不是 I/O 路径。DiskANN 是耦合的，不研究解耦的问题。
2. **能否指出现有工作的具体不足？** 能。DGAI 及类似系统的 reranking I/O 放大是可测量的。
3. **能否产生 ≥2× 改善？** 需要实验验证。如果解耦带来 2-3× I/O 放大（合理假设），消除大部分放大就能接近 2×。
4. **故事是否自洽？** 是。"高带宽 SSD 改变了 I/O trade-off → 解耦架构的额外 I/O 次数成为新瓶颈 → 系统设计" 链条完整。

### 2.6 与软件栈方向的关系

PZ 之前提出的软件栈方向和解耦架构方向**可以合并**：

- 软件栈方向的核心发现：SSD 利用率只有 24%（IISWC 2025）
- 解耦架构方向的核心问题：额外的 I/O 次数和路径序列化

两者的共同论点：**现代高带宽 SSD 下，图索引的 I/O 效率受限于软件路径和架构设计，而不是设备能力。** characterization 同时覆盖两个问题，系统设计可以同时解决两个问题。

### 2.7 风险

1. **改善幅度可能不够大。** 如果解耦架构的 I/O 放大只有 20-30%（不是 2-3×），改善后的差距太小。
2. **耦合架构已经存在（DiskANN）。** 如果结论是"耦合更好"，那只是证明了 DiskANN 的设计选择正确，不是新贡献。必须提出超越"简单耦合"的设计。
3. **CPU 计算可能主导。** 如果 beam search 的瓶颈是 PQ 距离计算或 reranking 计算，而不是 I/O，方向不成立。

## 3. 建议执行方式

不再使用之前的多轮 Kill gate 流程。改为：

1. **Claude 出一个简要的方向评估**（本文档）
2. **PZ 判断是否对这个方向有兴趣**
3. **Codex 做一个轻量级的 characterization pilot**（不是 prior-art audit）：在 DGAI 上 profiling beam search 的 I/O 分解，量化解耦架构的实际 I/O 放大
4. **基于 pilot 数据决定是否继续**

不在 pilot 之前做 exhaustive prior-art Kill。先看问题是否真实存在、改善空间有多大，再考虑论文 novelty 和 positioning。

这与 PZ 的建议一致："先痛批问题，再找故事。"
