## 1. RAG Document Revision Workload Evidence

### 1.1 Industry practice

Production RAG pipelines operate on a document-revision cycle, not a random-vector-insertion cycle:

- **Full re-indexing** remains common for small-to-medium knowledge bases: a daily cron job re-chunks, re-embeds, and replaces the entire vector store. Simple but expensive and causes downtime.
- **Incremental update** is the scaling target: event-driven ingestion triggered by source-system webhooks or git commit hooks. Only changed documents are re-chunked and re-embedded; unchanged chunks (identified by content hash) are skipped.
- **Semantic chunking** (structure-aware section/paragraph splitting) yields up to 70% higher retrieval accuracy than fixed-size windows, meaning production systems strongly prefer deterministic structure-aware chunking over arbitrary token windows.

The canonical update unit in a RAG knowledge base is:

```text
source document revision (e.g., one git commit touching one .md file)
→ deterministic re-chunking
→ content-hash exact matching to identify unchanged chunks
→ remove old changed/deleted chunks from index
→ insert new/modified chunks into index
```

A single document revision typically produces a **small group** (2–30 chunk-level updates) that are:
- derived from the same source text;
- generated by the same chunking policy;
- embedded by the same frozen model;
- therefore likely to cluster in embedding space.

### 1.2 Public revision corpora

Three categories of publicly available revision sources exist:

1. **Software documentation git repositories** — large projects like Linux kernel docs, Kubernetes docs, Rust book, Python docs have thousands of commits with per-file diffs recoverable via `git show`.
2. **API/systems documentation repositories** — AWS, GCP, Stripe, or OpenAPI spec repos with structured Markdown/YAML, frequent incremental edits.
3. **Wikipedia revision history** — MediaWiki API provides full revision content for any article; revision frequency and edit size distributions are well-studied.

All three allow deterministic recovery of old/new document content per commit, with complete provenance (commit hash, file path, timestamp).

### 1.3 Why the revision unit matters for graph ANN

When chunks from the same document revision cluster in embedding space, the graph ANN operations they trigger may overlap:

- **Graph search**: greedy search from the entry point traverses similar paths for nearby vectors, visiting many of the same intermediate nodes.
- **Neighbor candidates**: candidate sets from RobustPrune overlap because the same local neighborhood is being explored.
- **SSD pages**: if graph topology has any locality (as in Vamana/DiskANN's medoid-rooted structure), nearby vectors' adjacency lists may reside on the same or adjacent SSD pages.
- **Repair targets**: when deleting old chunks, reverse-edge repair must fix the same set of affected neighbors.
- **Write amplification**: modified adjacency pages may be the same pages, causing redundant read-modify-write cycles.

The core empirical question is whether this overlap is:
1. real and measurable;
2. larger than for random batches of the same size;
3. larger than for geometrically matched cross-document batches;
4. not already captured by trivial page caching or existing batch mechanisms.

### 2.7 Summary of gap

**No existing system exploits the document-revision group boundary at the graph-ANN index level.** The closest work is Slipstream, which exploits sequential stream locality for warm-starting insertion search, but:
- Slipstream has no explicit group boundary;
- Slipstream handles only insertion (no deletion/replacement);
- Slipstream does not deduplicate repair targets or page I/O;
- Slipstream does not use document identity.

The document-revision group is a stronger structural signal than mere stream locality: it provides an a-priori known set of related updates with predictable geometric clustering, enabling potential pre-computation of shared search state and repair-target deduplication.

## 3. Document-Group vs. Ordinary Batch: Strict Distinction

### 3.1 Ordinary batch

An ordinary batch update is a set of `g` vector insert/delete operations submitted together for efficiency:

```text
Batch = {u1, u2, ..., ug}
```

Properties:
- No semantic relationship between members required.
- Members may be geometrically distant.
- Batching motivation is throughput (amortize fixed costs, parallelize).
- Existing systems (FreshDiskANN consolidation, parallel HNSW insert) already implement this.

### 3.2 Geometric correlated batch

A batch where members happen to be geometrically close in embedding space:

```text
Geometric batch: pairwise distance or centroid distance ≤ threshold
```

Properties:
- Members cluster in embedding space → likely share graph search paths and SSD pages.
- No document identity or version relationship required.
- Could arise from any source: multiple documents on similar topics, synthetic data, etc.
- Any overlap benefit comes purely from embedding geometry.

### 3.3 Document-revision group

A group where all members are derived from a single document revision:

```text
Document group G:
  source = (document_id, commit_hash)
  members = {chunks changed between old and new version}
  each member has: old_chunk (to delete) and/or new_chunk (to insert)
```

Properties:
- Members share document provenance and version lineage.
- Members are likely geometrically close (same topic, similar text).
- Members have correlated insert/delete pairing (old chunk removed, new chunk inserted).
- Group boundary is known a priori from the application layer.
- Group size and composition are determined by the revision, not by a tuning parameter.

### 3.4 The critical experimental question

The hierarchy of explanatory power is:

```text
Document-revision group
⊇ Geometric correlated batch
⊇ Ordinary batch
⊇ Independent updates
```

The experiment must determine whether each level provides additional, measurable overlap benefit beyond the level below it. Specifically:

1. **Document group vs. geometric control**: Does document identity provide overlap beyond what geometric clustering alone predicts? (Tests whether version lineage, text continuity, or chunking structure adds signal.)
2. **Geometric batch vs. random batch**: Does geometric proximity alone produce measurable overlap? (Tests whether the phenomenon is purely geometric.)
3. **Real group vs. same-document shuffled control**: Does true revision lineage matter, or is "same document" sufficient? (Tests whether old↔new pairing contributes.)

## 4. Three Falsifiable Hypotheses

### H1: Read-Reuse Hypothesis

> Real document-revision groups produce significantly higher ReadReusePotential than size-matched random batches AND geometrically-matched cross-document batches, across at least two revision sources and two chunking policies.

**Falsified if:** ReadReusePotential(real group) ≤ ReadReusePotential(geometric control) + measurement noise, or if ReadReusePotential(real group) ≤ ReadReusePotential(random control) + measurement noise.

**Measurement:** For each group G and its paired controls A (random), B (geometric), C (same-doc shuffled):

```text
ReadReusePotential(G) = 1 - |∪_i P_i| / Σ_i |P_i|
```

where P_i is the set of SSD pages read during update i.

### H2: Repair-Target Overlap Hypothesis

> Real document-revision groups produce significantly higher RepairTargetReuse than both matched controls, and this overlap is not already captured by a simple group-local page cache applied to serial updates.

**Falsified if:** RepairTargetReuse(real group) ≤ RepairTargetReuse(geometric control), or if Serial-group-cache baseline already achieves ≥ 90% of the union oracle's repair-target deduplication.

**Measurement:**

```text
RepairTargetReuse(G) = 1 - |∪_i A_i| / Σ_i |A_i|
```

where A_i is the set of affected/reverse-edge target nodes for update i.

### H3: Cache-Residual Hypothesis

> After applying a standard group-local page cache to serial updates (Serial-group-cache baseline), real document-revision groups still retain measurable cost advantage over the cached random and geometric controls, indicating that the overlap structure provides benefit beyond what trivial caching captures.

**Falsified if:** The ratio `(Serial-cold - Serial-group-cache) / (Serial-cold - Union-oracle)` is ≥ 0.9 for all groups (meaning cache captures nearly all potential benefit), or if the residual after caching is equal across real groups and controls.

**This is the most important hypothesis.** If H3 is falsified, the entire direction reduces to "use a page cache during batch updates," which is a known technique with no novelty.

## 5. Data Sources, Chunking, Embedding, and Paired Controls

### 5.1 Revision sources (preregistered)

Three sources, selected for diversity and reproducibility:

| # | Source | Type | Selection criteria |
|---|--------|------|--------------------|
| 1 | **kubernetes/website** | Large software documentation | >10,000 commits; Markdown files; active maintenance; Apache-2.0 license; well-structured sections |
| 2 | **rust-lang/book** or **python/cpython Doc/** | Language/API documentation | Thousands of commits; structured .md or .rst files; CC/PSF license; distinct document structure from source 1 |
| 3 | **Wikipedia revision subset** | Encyclopedia | MediaWiki API provides full revision content; fundamentally different document structure (prose vs. technical docs); CC-BY-SA license |

Selection is preregistered: repositories are chosen by commit volume and structural diversity, NOT by observing overlap metrics first.

### 5.2 Revision sampling

For each source:
1. Enumerate commits that modify exactly one document file (to isolate single-document revisions).
2. Recover old and new file content via `git show` or MediaWiki API.
3. Apply deterministic chunking to both versions.
4. Content-hash exact matching: chunks with identical content hash are excluded from the update group.
5. Remaining chunks form the group: deleted chunks (in old but not new), inserted chunks (in new but not old), modified chunks (content changed).

Preregister a fixed count of revision groups per source (e.g., 200 per source, stratified by group size 2–5, 6–15, 16+).

### 5.3 Chunking policies (two, preregistered)

1. **Structure-aware section/paragraph chunking**: Split on Markdown headers (##, ###) or reStructuredText section boundaries. Each section is one chunk. Sections exceeding a maximum token count (e.g., 512 tokens) are further split at paragraph boundaries.

2. **Fixed-token-window robustness policy**: Non-overlapping windows of 256 tokens. No structure awareness. This policy serves as a robustness check: if overlap results depend entirely on the chunking policy, the phenomenon is an artifact of chunking, not of document revision structure.

Both policies are frozen before any overlap measurement.

### 5.4 Embedding models (two, CPU-compatible)

1. **all-MiniLM-L6-v2** (384-dim, Sentence-Transformers): Widely used, fast on CPU, well-characterized.
2. **nomic-embed-text-v1.5** (768-dim, Nomic AI): Recent, higher quality, still CPU-feasible.

Requirements:
- Model revision/hash pinned at experiment start.
- L2 distance metric, vectors normalized to unit length.
- No LLM, no API, no GPU.
- Embedding generation time reported separately (not included in ANN update cost).

### 5.5 Paired controls (three, per group)

For every real revision group G of size g:

**Control A (Random batch):**
- Sample g vectors uniformly from the active corpus.
- Match insert/delete ratio and chunk-length distribution.
- Sample from the same active-set checkpoint as G.

**Control B (Geometric cluster, cross-document):**
- Select g vectors from different source documents whose centroid distance and pairwise distance distribution match G's.
- All document IDs must differ from G's.
- Match insert/delete ratio.
- This control isolates the geometric clustering effect from document identity.

**Control C (Same-document, shuffled revision):**
- From the same source document as G, select chunks from a non-adjacent revision (≥5 commits apart).
- Or: shuffle the old↔new chunk pairing within G, breaking the true revision lineage while preserving document identity.
- This control tests whether revision lineage (not just document identity) matters.

All controls are generated and frozen before any update trace is collected.

### 5.6 Index baselines (four conditions)

Each group and its controls are applied to identical frozen index states:

1. **Serial-cold**: Independent per-vector updates, no group cache. Each update's page reads start cold.
2. **Serial-group-cache**: Same algorithm, but a group-local page cache retains pages read by earlier updates in the same group. No shared candidates, no modified prune.
3. **Existing batch/buffered path**: Run the system's native batch mechanism (if available) on the group. E.g., FreshDiskANN's StreamingMerge/consolidation, DGAI's native batch, etc.
4. **Union oracle**: Upper bound only. Assumes each distinct page is read/written at most once across the group, but all logical work (search, prune) is preserved.

### 5.7 Index systems

At least two disk-resident dynamic graph ANN paths:
- **DGAI-style**: Uses PZ's existing DGAI implementation.
- **OdinANN-style**: Uses PZ's existing OdinANN implementation.

Optionally: IP-DiskANN or topology-aware localized update as a third boundary (if installable without excessive effort).

Each group starts from the same frozen index fork: `hash(index_state)` verified identical across all conditions.

## 6. Strongest Novelty Objections

### Objection 1: "This is just page caching during batch updates"

**Severity: HIGH.** If Serial-group-cache captures ≥90% of the union oracle benefit, then the only mechanism is "cache pages read by earlier updates in the batch." This is a standard OS/application-level technique with zero novelty. H3 directly tests this.

**Mitigation:** The experiment must measure what fraction of overlap is captured by the cache and what fraction remains. If repair-target overlap or candidate-set sharing provides additional benefit beyond page-level caching, the objection is partially addressed. But if page cache dominates, the direction is DEAD.

### Objection 2: "Geometric proximity, not document identity, drives all overlap"

**Severity: HIGH.** If Control B (geometric cluster from different documents) shows the same overlap as real groups, then document identity provides no additional signal. The phenomenon is purely geometric, and the "RAG document group" framing is misleading.

**Mitigation:** Control B directly tests this. If document groups ≈ geometric controls, the direction pivots to "correlated geometric batch update" (HOLD-GEOMETRIC-CORRELATION-ONLY per gate), which requires a separate gate.

### Objection 3: "Existing batch/localized update mechanisms already capture the benefit"

**Severity: MEDIUM-HIGH.** Topology-aware localized update already co-locates repair within a bounded neighborhood. FreshDiskANN consolidation already batches deletions. If these mechanisms, applied to a group, already achieve near-oracle deduplication, there is no residual for a new mechanism.

**Mitigation:** The experiment must run the existing batch path on the same groups and controls. If the existing batch path already captures most of the benefit, the direction is KILLED (KILL-CACHE-OR-BATCH-ABSORBS-GAIN).

### Objection 4: "Group sizes are too small to matter"

**Severity: MEDIUM.** If typical document revisions produce groups of 2–5 chunks, the absolute I/O savings from deduplication may be negligible compared to the overhead of group detection, control generation, and bookkeeping.

**Mitigation:** Stratify results by group size. Report absolute savings (bytes, pages, microseconds), not just relative overlap percentages. If small groups dominate and absolute savings are negligible, the direction may be killed even with positive relative metrics.

### Objection 5: "Slipstream's warm-start already captures the insertion benefit"

**Severity: MEDIUM.** Slipstream shows that consecutive insertions of nearby vectors can share search state via warm-starting. If document-group inserts are processed sequentially, Slipstream's mechanism (or a similar warm-start) may capture all the search-path overlap.

**Mitigation:** Slipstream handles only insertion, not deletion/replacement. Document revisions involve paired delete-old/insert-new operations. Slipstream does not deduplicate repair targets (it has no repair) or modified pages. However, if the deletion component shows no group-specific overlap, this objection significantly weakens the direction.

### Objection 6: "The direction lacks a clear optimization target"

**Severity: MEDIUM.** A0 is measurement-only. Even if overlap is measured and confirmed, the direction has no defined mechanism (by gate design). A reviewer may ask: "You measured overlap — so what? What algorithm exploits it?" This is acceptable for A0 (profiling gate) but becomes a showstopper if A0 passes and no mechanism emerges.

**Not addressed in A0 by design.** The gate explicitly prohibits proposing mechanisms.

## 7. Preliminary Assessment: Should We Proceed to Profiling?

### Arguments FOR proceeding

1. **Genuine gap:** No existing system exploits document-revision group boundaries at the graph-ANN level. The closest work (Slipstream) is insertion-only, opportunistic, and has no explicit group concept.

2. **Structurally sound hypotheses:** All three hypotheses are cleanly falsifiable with matched controls. The experiment design is rigorous and preregistered.

3. **Low cost:** A0 profiling requires only existing DGAI/OdinANN infrastructure, CPU-compatible embeddings, and public git repositories. No GPU, no new system implementation, no large dataset download.

4. **High-value outcome either way:** If H1–H3 are confirmed, a new and well-motivated direction opens. If they are falsified, the community gains a clean negative result about document-group structure in graph ANN — useful for ruling out this design space.

5. **Real workload relevance:** RAG document updates are a dominant use case for vector databases. If group structure can be exploited, the impact is immediately practical.

### Arguments AGAINST proceeding

1. **Cache-absorption risk is high:** The most likely outcome is that Serial-group-cache captures most of the overlap, leaving no room for a novel mechanism. Page caching is very effective for correlated access patterns, and graph ANN page access patterns for nearby vectors are exactly the kind of pattern caches excel at.

2. **Group sizes may be too small:** Typical document revisions change 2–10 chunks. At this scale, even substantial relative overlap (e.g., 50% ReadReusePotential) translates to saving only a few page reads — potentially not worth the bookkeeping.

3. **No mechanism in sight:** Even if overlap is confirmed, converting it into an algorithm that beats Serial-group-cache is a separate, unsolved problem. Combined prune has unknown semantics; shared frontier is not in the literature.

### Recommendation

**PROCEED to A0 profiling with explicit fail-fast gates.**

The direction has a genuine literature gap, the hypotheses are cleanly testable, the cost is low, and the outcome is informative regardless of result. However, the fail-fast condition must be strict:

- If Serial-group-cache captures ≥ 85% of union-oracle benefit → KILL immediately, do not attempt mechanism design.
- If real groups ≈ geometric controls on all metrics → HOLD (geometric correlation only, not document-group).
- If absolute savings for median group size < 5 pages read or 2ms wall time → KILL (too small to matter).

The profiling is worth doing because it answers a clean empirical question at low cost. But the prior probability of KILL is, in my assessment, **≥ 60%**, primarily due to cache-absorption risk (Objection 1).

## Appendix A: Trace Schema (per update)

```text
dataset_id                    # which revision source
document_id                   # source document path
commit_hash                   # git commit or Wikipedia revision ID
revision_id                   # unique revision group identifier
chunk_id                      # chunk identifier within group
chunk_content_hash            # SHA-256 of chunk text
chunk_operation               # INSERT / DELETE
embedding_hash                # SHA-256 of embedding vector bytes
group_id                      # revision group ID
group_ordinal                 # position within group (0-indexed)
group_size                    # total members in group
control_type                  # REAL / CONTROL_A / CONTROL_B / CONTROL_C
graph_node_id                 # assigned node ID in graph index
visited_node_ids              # set of nodes visited during greedy search
candidate_node_ids            # set of candidate nodes from search
page_ids_read                 # set of SSD pages read (with file role)
page_read_timestamps          # per-page issue/completion times
affected_node_ids             # reverse-edge / repair targets
prune_input_candidates        # RobustPrune input set
prune_output_neighbors        # RobustPrune output set
modified_page_ids             # pages whose content was modified
written_page_ids              # pages submitted for write
logical_bytes_read            # application-level read bytes
logical_bytes_written         # application-level write bytes
physical_bytes_read           # I/O engine / cgroup bytes
physical_bytes_written        # I/O engine / cgroup bytes
index_system                  # DGAI / OdinANN / other
baseline_condition            # serial-cold / serial-group-cache / existing-batch / union-oracle
chunking_policy               # structure-aware / fixed-token-window
embedding_model               # model identifier + hash
index_fork_hash               # hash of frozen index state before group
```

Closure requirement:
```text
Σ application page reads = Σ I/O engine submitted reads = Σ device/cgroup completed read bytes / page_size
```

Any discrepancy must have an explicit merge/cache explanation with zero unexplained residual.
