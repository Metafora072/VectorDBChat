## Filtering Criteria Applied

- Eliminated ideas requiring GPU or distributed infrastructure

## Top 8 Survivors (Ranked)

### T1: Semantic Write Amplification of Agent Memory
**Source**: Codex #8
**Summary**: Quantify the cascading physical cost when one semantic revision triggers rewrites across text, embeddings, provenance, conflict edges, and cached plans. Use evidence/interpretation separation to suppress it.
**Hypothesis**: Revision, merge, and forgetting operations have physical write amplification far exceeding simple insertion; immutable evidence objects with lightweight interpretation versions and epoch consolidation can significantly reduce writes.
**Why not "apply X to Y"**: Amplification source is not LSM compaction but multiple randomly-derived representations of the same memory, plus model/prompt upgrades invalidating interpretation versions.
**MVP**: Implement slim GEM-style operation layer, use existing M0-M3 write attribution + page lifecycle on XFS/multi-NVMe. Compare whole-object rebuild vs separated layout.
**Contribution**: empirical finding / system | **Risk**: LOW | **Effort**: 3-5 weeks | **Venue**: FAST
**PZ fit**: **HIGHEST** — directly reuses M0-M3 infrastructure on a genuinely new workload.

### T2: Closed-Loop Working Set Phase Transition
**Source**: Codex #1
**Summary**: Measure whether agent memory capacity reduction causes nonlinear performance collapse through the feedback loop: recall change → action change → new data change.
**Hypothesis**: Agent cache miss doesn't just add latency — it changes subsequent access distribution. Below a capacity threshold, task success rate may suddenly drop with recovery hysteresis.
**Why not "apply X to Y"**: Traditional caches face exogenous request streams; agent request streams are endogenously generated by previously recalled content. Fixed-trace replay hides this feedback loop.
**MVP**: Scripted agent with cached model outputs on 30-50 long tasks, sweep DRAM capacity and LRU/LFU/semantic-heat policies. Compare real closed-loop vs fixed-trace replay. Measure success rate, access drift, NVMe I/O, hysteresis.
**Contribution**: empirical finding / theoretical | **Risk**: MEDIUM | **Effort**: 4-6 weeks | **Venue**: FAST/EuroSys/NeurIPS

### T3: External Side-Effect Receipts & Recoverability Envelope
**Source**: Codex #4
**Summary**: For non-idempotent tool calls, record detectable effect receipts and formalize which crash states are recoverable, compensable, or fundamentally undecidable.
**Hypothesis**: The intent→dispatch→effect→persist ambiguity window can be significantly narrowed by nonce, effect fingerprint, and recovery probes, but some APIs have irreducible lower bounds in post-effect/pre-record states.
**Why not "apply X to Y"**: Agent combines non-resumable streaming generation AND non-rollbackable real-world side effects. WAL/workflow replay cannot determine whether an unreturned call has taken effect.
**MVP**: Implement 8-12 local/mock tools covering idempotent, queryable, compensable, and undetectable properties. Inject crashes at each micro-boundary. Compare checkpoint, blind retry, and receipt protocol on duplicate/loss/ambiguity rates.
**Contribution**: theoretical / system | **Risk**: MEDIUM | **Effort**: 5-8 weeks | **Venue**: OSDI/EuroSys

### T4: Incremental Code Understanding Store with Support Anchors
**Source**: Codex #10 + Claude
**Summary**: Record symbol, call-edge, test, and assumption anchors for persistent code summaries so commits only invalidate truly affected agent knowledge.
**Hypothesis**: File-level rebuild wastes severely; AST-dependency-only preserves semantically stale conclusions. Explicit support anchors achieve better staleness-rebuild tradeoff.
**Why not "apply X to Y"**: Compilation artifacts have complete, deterministic dependency declarations. Agent-generated NL knowledge contains implicit assumptions; code compiling doesn't mean "this lock guards that field" is still true.
**MVP**: 5-10 large repos, ~1000 historical commits, fixed code questions with cached summaries. Compare full rebuild, file-level, symbol-level, and support-anchor invalidation on stale answer rate, rebuild bytes, NVMe I/O.
**Contribution**: empirical / system | **Risk**: MEDIUM | **Effort**: 6-8 weeks | **Venue**: OSDI/VLDB/EuroSys

### T5: Verifiable Forgetting with Semantic Deletion Closure
**Source**: Codex #12
**Summary**: When deleting original memory, locate all derived summaries, beliefs, cached contexts, trace snapshots, and experience graph nodes still influenced by it. Generate a forgetting certificate.
**Hypothesis**: Tombstones leave massive semantic residue; full ancestor closure over-deletes. Recording minimal support sets and enforcing deletion epoch fences at retrieval boundaries achieves verifiable completeness-retention tradeoff.
**Why not "apply X to Y"**: Agent data propagates through non-deterministic, multi-source, irreversible model transformations. Correct target is not deleting byte copies but preventing deleted information from influencing future actions.
**MVP**: Inject 10K traceable canary facts plus their rewrites, summaries, multi-hop inferences into a full memory pipeline. Compare record deletion, full closure, minimal support set, and full recomputation on semantic residue, over-deletion, recompute I/O, write amplification.
**Contribution**: new method / system | **Risk**: MEDIUM | **Effort**: 5-7 weeks | **Venue**: VLDB/FAST

### T6: Causal Belief Snapshot
**Source**: Codex #3
**Summary**: Guarantee that one retrieval sees raw evidence, summaries, contradiction judgments, and index entries from compatible ancestor versions — not "old evidence + new conclusion."
**Hypothesis**: Even if all records satisfy the same MVCC timestamp, async summarization and index updates produce unrealizable "old evidence + new conclusion" views. A compact ancestor frontier eliminates this.
**Why not "apply X to Y"**: Consistency constraints live in a random, async semantic derivation graph, not in record commit timestamps.
**MVP**: Build ingestion→revision→summary→index pipeline, enumerate and randomize interleavings. Compare read-committed, MVCC, and ancestor-frontier retrieval on inconsistent answer rate, metadata overhead, throughput.
**Contribution**: theoretical / system | **Risk**: MEDIUM | **Effort**: 5-7 weeks | **Venue**: VLDB/OSDI

### T7: Decision-Regret-Driven Memory Admission
**Source**: Codex #7
**Summary**: Use counterfactual replay to estimate action regret from deleting a candidate memory, rather than using frequency, similarity, or confidence.
**Hypothesis**: Truly important memories are often low-frequency failures, counter-evidence, and constraints. A regret proxy learned from historical trajectories outperforms generic hotness at the same capacity.
**Why not "apply X to Y"**: Agent writes are often model-generated summaries/reflections; incorrect memories in a feedback loop compound. Object value depends on future behavior, not future reads.
**MVP**: Generate ~10K episodes on code repair / interactive tasks. Offline mask individual memories to get oracle regret. Train lightweight CPU model. Compare FIFO, reservoir, confidence, and learned admission.
**Contribution**: empirical / method | **Risk**: HIGH | **Effort**: 5-7 weeks | **Venue**: ICML/NeurIPS/VLDB

### T8: Prospective Memory Trigger Index
**Source**: Codex #6
**Summary**: Compile "notify me when CI passes" type prospective memory into revisable, interpretable, exactly-once persistent standing queries.
**Hypothesis**: Rule-filter + semantic-candidate + state-verification three-tier index avoids full scans and blocks erroneous triggers when intent is revised or its preconditions invalidated.
**Why not "apply X to Y"**: Agent intents are NL-generated, matching depends on evolving context and memory. Event similarity ≠ trigger condition satisfaction; triggering itself changes agent state.
**MVP**: Generate 100K intents and 10M events with CPU embeddings. Compare scan, vector-first, and three-tier index on F1, p99, I/O, duplicate triggers, and revision correctness.
**Contribution**: method / system | **Risk**: MEDIUM | **Effort**: 4-6 weeks | **Venue**: VLDB/ICLR

## Claude's Assessment
T1 is the strongest starting point: it directly leverages PZ's unique M0-M3 infrastructure on a genuinely novel workload. If semantic write amplification is significant, it opens a research program (T5, T6 become follow-ups). If it's not significant, that's a publishable negative result that settles whether agent memory needs a custom storage engine.
