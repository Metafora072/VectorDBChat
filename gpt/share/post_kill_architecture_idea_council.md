# 驻盘图索引研究重置：Architecture Idea Council

## 当前正式结论

以下研究平面已经关闭：

* 动态 insert 路径的局部成本优化；
* topology write 与 append-only adjacency；
* coordinate acquisition 与 exact-vector access；
* PQ、heap、visited 等 search-loop 小项；
* 统一维护债务指标；
* query/update locality transfer；
* write-set constrained relayout；
* 跨系统 query/update SSD 干扰；
* 以 libaio→io_uring、SPDK、fixed files、registered buffers、pipeline width 或跨查询协程为核心的软件路径优化。

最后一项的关闭依据是 PipeANN、NAVIS、VeloANN 和 Turbocharging 已覆盖主要机制。PZ 关于现代 SSD 下软件开销重要的判断仍可作为系统背景，但不能再作为当前工作的 novelty 主张。

下一阶段不再围绕上述平面换名字或组合机制。

---

## 研究约束

新候选必须满足：

* 单机环境；
* 可以使用多块本地 NVMe；
* 不依赖 GPU、RDMA、分布式集群或 CXL；
* 不以崩溃恢复为核心贡献；
* 不以普通缓存、优先级队列、限速、batching、固定阈值或接口替换为主要创新；
* 不依赖“首次把通用技术 X 用到 ANN”；
* 贡献必须是系统架构或新的系统抽象；
* 有真实负载或可下载数据；
* 一周内可以完成低成本 Problem Gate；
* 实现工作量足以支撑毕业设计，但不能成为大量分散的工程修补。

# 第一阶段：Claude 独立提出架构假设

Claude 从“现有架构的隐含假设在什么条件下失效”出发，而不是继续寻找函数级瓶颈。

最多提出四个候选；没有合格候选时可以明确提交空列表。

每个候选必须包含：

## 1. 被挑战的架构假设

该假设必须存在于至少三个代表性系统或一类主流设计中，例如：

* 索引生命周期假设；
* DRAM/SSD 资源比例假设；
* 单索引独占资源假设；
* 构建、查询、更新相互分离的假设；
* 数据分布长期稳定的假设；
* 单盘或统一存储介质假设；
* 维护可以离线或集中完成的假设。

不能只写“现有系统较慢”。

## 2. 真实失效场景

说明：

* 哪类应用或部署会违反假设；
* 为什么不是人为构造；
* 哪些现有系统会因此失败；
* 失败表现是吞吐、尾延迟、内存、构建时间、写放大、空间、可用性还是质量。

## 3. 架构级桥梁假设

必须描述一种新的组织方式、执行模型或存储抽象，而不是直接罗列优化组件。

合格形式：

> 因为现有系统把 A 和 B 固定绑定，场景 C 下产生结构性代价；若将其重构为 D，并维护不变量 E，可能同时消除代价 F。

不合格形式：

* 使用 io_uring；
* 增加缓存；
* 动态调整 beam；
* 添加预取；
* 根据阈值切换；
* 将几个已有机制组合。

## 4. 最接近工作与 novelty 边界

至少给出：

* 三项最接近工作；
* 哪一项最可能直接覆盖候选；
* 候选和它们的结构性区别；
* 审稿人最强拒稿理由。

必须使用论文原文或官方 artifact，不依赖二手总结。

## 5. 最小 Kill 实验

只设计一个低成本实验，回答最关键的存在性问题。

必须明确：

* 输入；
* baseline；
  -观测指标；
* 通过条件；
* 立即 Kill 条件；
* 所需时间、磁盘和代码工作量。

## 6. 系统论文形态

预先说明候选若成立，最终论文至少需要什么：

* 一个核心机制还是多个部件；
* 核心不变量；
* 理论或成本模型；
* 端到端系统；
* 哪些实验是不可缺少的。

Claude 输出：

```text
claude/share/post_kill_architecture_hypotheses_0712.md
```

# 第二阶段：Codex 独立对抗审查

Codex 在 Claude 发布后再开始，不提前共同润色候选。

对于每个候选，独立完成：

## Prior-art 攻击

* 搜索 2024–2026 一手论文和官方代码；
* 检查是否已有同样问题和机制；
* 区分正式论文、预印本、artifact 与推测；
* 特别检查 Claude 的系统类型、索引类型和机制描述是否准确。

## 架构可实现性攻击

* 当前服务器是否能实现；
* 是否需要不可用硬件；
* 是否需要重写完整索引；
* 是否依赖论文未公开代码；
* 是否存在隐藏的元数据、恢复或一致性负担；
* 是否只是把工作移动到后台或其他阶段。

## 问题真实性攻击

* 是否能找到真实 workload；
* 问题是否至少在两个系统或两类架构中出现；
* 是否只有极端参数下出现；
* 简单 baseline 是否已经足够；
* 最优可能收益是否足以支撑端到端论文。

## 裁决

对每个候选给出：

```text
KILL
REVISE
PROVISIONAL
```

`PROVISIONAL` 只表示值得进入最小 Problem Gate，不代表立项。

Codex 输出：

```text
codex/share/post_kill_hypothesis_adversarial_review_0712.md
```

本阶段不修改系统、不运行正式实验。

# 第三阶段：Gpt 统一裁决

Gpt 对 Claude 与 Codex 的结果分别审查，不因 Claude 的创造性或 Codex 的可实现性单独放行。

每个候选按以下维度评分：

| 维度                | 权重 |
| ----------------- | -: |
| 问题真实性与场景价值        | 15 |
| 架构级 novelty       | 20 |
| 系统味道              | 15 |
| 与 prior art 的清晰边界 | 15 |
| 可证伪性与因果闭环         | 10 |
| 当前环境可实现性          | 10 |
| 端到端收益上界           |  5 |
| 工作量与毕业适配          |  5 |
| 真实负载与可复现性         |  5 |

自动 Kill 条件：

* novelty 依赖接口或硬件替换；
* 只有一个系统存在；
* 最强 baseline 是普通通用技术；
* 需要多个小优化才能形成贡献；
* 问题无法用低成本实验证伪；
* 成功后仍只能写成测量论文；
* 需要不可用的分布式/GPU/RDMA 环境；
* 与已关闭平面只是命名差异。

最多批准两个候选进入 Problem Gate，优先只批准一个。

# 候选搜索视野

以下仅作为搜索视野，不是预设方向：

* 十亿级索引的构建、重建与在线演化；
* 低 DRAM 条件下的资源比例与索引可运行性；
* 单机多 NVMe 上的索引组织，而不是普通磁盘条带化；
* 多索引或多租户共享存储时的结构性冗余；
* 数据分布和查询分布变化下的索引生命周期；
* 查询、构建、更新与维护之间尚未被重构的阶段边界；
* 图索引之外但仍属于向量搜索存储路径的架构问题。

任何候选都必须从事实和失效场景出发，不能从“我们会什么技术”反推问题。

# 协作纪律

* Claude负责提出少而强的假设，不负责替候选辩护；
* Codex负责尽可能杀死候选，不因实现方便而保留；
* Gpt负责门禁和因果闭环，不做折中拼装；
* PZ负责资源、研究兴趣和最终立项；
* 阴性结论应被保留，不重复验证；
* 一个强候选优于四个半成品；
* 没有合格方向也是允许的正式结论。
