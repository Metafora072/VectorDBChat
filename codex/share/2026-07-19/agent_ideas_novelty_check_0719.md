## T1: Semantic Write Amplification of Agent Memory

**Verdict**: INCREMENTAL | **Novelty Score**: 4/10

**Closest prior work**:
- **Eywa** (arXiv:2605.30771): Almost verbatim covers "evidence/interpretation separation" — immutable source evidence as authoritative base, canonical LLM-derived beliefs as revisable index.
- **Kumiho** (arXiv:2603.17244): Immutable revisions, `Depends_On`/`Derived_From`/`Supersedes`, `AnalyzeImpact` cascade, async consolidation. Explicitly describes traversing all downstream conclusions for re-evaluation after belief update.
- **GEM** (arXiv:2605.26252): Defines agent memory with revision, dependency-aware propagation, provenance, derived salience. Proposes I/O-aware layout, embeddings, typed dependencies.
- **TierMem** (arXiv:2602.17913): Immutable raw log, provenance-linked summaries, verified write-back consolidation.
- **Agent Memory Characterization** (arXiv:2606.06448): Already measured construction latency, LLM/embedding traffic, energy, storage footprint. Graph memory amplifies embedding traffic. But did NOT define "per-revision semantic write amplification."
- **CASCADEKG** (ACL Findings 2026): Formalized local fact updates causing multi-hop cascading updates in KG.

**Genuine differentiation**:
- Defining a repeatable "amplification factor" metric: attributing one logical revision to physical operations across text, embedding, provenance, conflict edges, plan caches, summaries
- Model/prompt upgrade as interpretation invalidation source
- Epoch consolidation reducing actual LLM calls, embedding rewrites, DB writes, I/O

**Risk**: "immutable evidence + revisable interpretation" is NOT new (Eywa). Dependency propagation covered by GEM/Kumiho/CASCADEKG. Reviewers will likely see this as "incremental view maintenance cost measurement."

**Recommendation**: Proceed cautiously. Reposition as **measurement/characterization paper** — define the metric, measure across systems, then optimize. Do NOT claim evidence/interpretation separation as novel architecture.

## T2: Closed-Loop Working Set Phase Transition

**Verdict**: INCREMENTAL | **Novelty Score**: 5/10

**Closest prior work**:
- **MemoryArena** (arXiv:2602.16313): Explicitly uses Memory–Agent–Environment closed loop; criticizes static recall benchmarks that separate memorization from action. "Fixed trace hides feedback" cannot be claimed as first discovery.
- **Memory in the Loop** (arXiv:2607.05690): Causal intervention — fixed memory-latency budget, varied store latency, observed redundant actions increase with latency. Directly proves memory system state changes agent behavior.
- **Pichay** (arXiv:2603.09023): Models context as working-set cache with demand paging, fault-driven pinning. Observed thrashing under extreme pressure. But core results rely on offline replay.
- **ACL 2026** "How Memory Management Impacts LLM Agents" (aclanthology.org/2026.acl-long.27/): Experience-following, incorrect memories affect current behavior and pollute future via write-back. Capacity-constrained memory vs performance curves — reports **gradual convergence, NOT phase transition**.
- Traditional closed queueing/cache networks: "Closed" means requests circulate, not that request semantics are generated by cache hits. Weakens terminology novelty but not a direct scoop.

**Genuine differentiation**:
- Cache capacity/miss outcome as **online causal variable** (not just memory quality or retrieval latency)
- Demonstrating **non-smooth critical points**, bistability, and hysteresis (increase vs decrease capacity paths differ)
- Building closed-loop model: cache state → recalled content → action → future request distribution → controller design

**Risk**: "Endogenous request streams" already established by MemoryArena + Memory in the Loop + ACL 2026. Remaining novelty depends entirely on whether phase transition and hysteresis actually exist. If experiments show only smooth degradation or model noise, the idea loses its main contribution.

**Recommendation**: Proceed with caution. Run small-scale bidirectional capacity sweep first. Only invest in system implementation after reproducibly demonstrating hysteresis and ruling out randomness/task hard thresholds.

## T3: External Side-Effect Receipts & Recoverability Envelope

**Verdict**: **SCOOPED** | **Novelty Score**: 2/10

**Closest prior work**:
- **Classic RPC**: RIFL (unique RPC ID + durable completion records for exactly-once)
- **Beldi** (OSDI 2020): Log + transaction for fault-tolerant stateful serverless workflows
- **ExoFlow** (OSDI 2023): Already classifies recovery strategies by determinism, external visibility, rollbackability
- **ACRFence** (arXiv:2603.20625): Agent-specific — records irreversible tool effects, replay-or-fork on checkpoint restore. Handles LLM generating semantically equivalent but bytewise different requests (breaking standard idempotency)
- **DART** (arXiv:2605.23311): Formalizes semantic recoverability boundary for structured tool agents; determines legal recovery points or refuses recovery based on dependency/effect constraints
- **NabaOS** (arXiv:2603.10060): HMAC-signed tool receipts
- Temporal/LangGraph/Azure Durable Functions: All acknowledge at-least-once for side-effecting activities

**Remaining space**: Only a very narrow formalization of ambiguity window reducibility lower bounds across real API properties.

**Recommendation**: **KILL**. Core problem, receipt/dedup mechanism, agent checkpoint with irreversible effects, and recoverable boundary all have direct prior work. ACRFence specifically handles the agent-specific combination.

## T4: Incremental Code Understanding Store with Support Anchors

**Verdict**: INCREMENTAL | **Novelty Score**: 3/10

**Closest prior work**:
- **RepoDoc** (arXiv:2604.26523): Most direct competitor — persistent RepoKG with Code/Concept/Doc entities, bidirectional semantic impact propagation after code changes, selective regeneration. Already reports update recall, time, and token savings.
- **Completion Semantics** (arXiv:2607.12490): Explicitly identifies that LLM code conclusions depend on implicit omitted-context assumptions; uses executable witnesses to expose required completion assumptions.
- **JTMS/ATMS** (1980s): Per-conclusion justification/support sets and assumption environments with dependency-directed retraction.
- Incremental static analysis: Mature dynamic dependency tracking, lazy invalidation.

**Genuine differentiation**:
- Support relations attached to each NL conclusion (not just documents, modules, or code entities)
- Anchor types covering symbol, call edge, test outcome, negative/omitted-context assumption
- Need real code evolution benchmark with human labels for "conclusion still true / stale"

**Risk**: Direct combination of RepoDoc + Completion Semantics + TMS. Reviewers will see support anchors as "finer edge schema" for RepoDoc. Automatic assumption anchor extraction reliability is unproven.

**Recommendation**: Proceed cautiously. Must abandon "first incremental code understanding" positioning. Reframe as "conclusion-level support extraction accuracy and staleness-cost frontier."
